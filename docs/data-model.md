# Data Model (V1)

The system uses a **ledger-based model**, separating transactional events from balance representation.

---

## accounts

Stores account identity and configuration.

- id
- account_number
- branch_number
- bank_code
- holder_name
- account_type
- currency
- status
- allow_negative_balance
- overdraft_limit
- created_at
- updated_at

---

## transactions

Represents a business transaction that generates one or more ledger entries.

### Fields

- transaction_id
- source_system
- external_transaction_id
- transaction_type (deposit, withdraw, transfer)
- occurred_at
- recorded_at

### Semantics

- `transaction_id`: internal identifier generated by the service for the business transaction
- `source_system`: external system that originated the transaction
- `external_transaction_id`: transaction identifier provided by the external source
- `transaction_type`: business transaction type
- `occurred_at`: when the event happened in the business domain (provided by the payload when available)
- `recorded_at`: when the transaction was recorded by the system

### Notes

- All timestamps are stored in UTC (`Instant`)
- (`source_system`, `external_transaction_id`) should be unique to support idempotency
- A transaction may generate one or more ledger entries

---

## ledger_entries

Represents immutable financial postings for a single account impact.

Each business transaction generates one or more ledger entries.

### Fields

- id
- transaction_id
- account_id
- entry_type (credit, debit)
- amount
- occurred_at
- recorded_at

### Semantics

- `transaction_id`: internal identifier that links the ledger entry to its parent transaction
- `account_id`: account impacted by the ledger entry
- `entry_type`: posting direction for the account impact (`credit` or `debit`)
- `amount`: signed amount stored according to the posting effect
    - positive for credits
    - negative for debits
- `occurred_at`: when the event happened in the business domain
- `recorded_at`: when the ledger entry was recorded by the system

### Notes

- All timestamps are stored in UTC (`Instant`)
- Ledger entries are immutable once created
- Each ledger entry represents a single account impact
- Each ledger entry belongs to exactly one transaction
- A transfer should generate two ledger entries under the same `transaction_id`
- For each `transaction_id`, the sum of all `amount` values should be zero

---

## daily_account_balances

Represents the daily materialized balance snapshot of each account, derived from ledger entries.

- id
- account_id
- balance_date
- opening_balance
- total_credits
- total_debits
- closing_balance
- created_at
- updated_at

### Rules

- `total_credits` is the sum of all positive ledger amounts for the day
- `total_debits` is the absolute sum of all debit ledger amounts for the day
- `closing_balance = opening_balance + total_credits - total_debits`
- next day opening_balance = previous closing_balance

## Design Decisions

- ledger entries are immutable
- balance is derived, not stored as a single field
- daily snapshots improve read performance
- limits are separated from account configuration
- transactional context is separated from account postings
- business transactions must remain balanced in the ledger
